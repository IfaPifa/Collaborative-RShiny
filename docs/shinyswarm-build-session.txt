3. INITIAL IMPLEMENTATION
--------------------------------------------------------------------------------

  R/capture_state.R, R/restore_state.R, R/wrappers.R, R/utils.R
  inst/examples/calculator/app.R

Key design points implemented:
  - Transport-agnostic: functions emit/consume JSON only.
  - Reactives registered by name through ... (R cannot auto-discover them).
  - Typed dispatch in .capture_one / .restore_one.
  - digits = NA in toJSON to preserve full numeric precision (critical for
    exact reproducibility of non-deterministic results).

First test run: 1 failure - JSON rounded a Monte Carlo value to 4 decimals.
Fix: digits = NA. All tests passed afterward (33 checks).
Calculator demo booted and served HTTP 200 locally.

4. CODE REVIEW #1 (skill: "Code Review like Bart")
--------------------------------------------------------------------------------
Findings (most important = security, because cross-user restore = untrusted input):
  [High]      Path traversal via attacker-controlled filename (PoC confirmed:
              filename "../ESCAPED.txt" wrote outside target dir).
  [Med/High]  unserialize() on untrusted checkpoint data (code-exec risk).
  [Medium]    reactiveValues restore was additive (union), not exact -> cross-
              contamination risk; also untested.
  [Low]       Undeclared stats:: dependency.
  [Minor]     null round-trip behaviour; friendlier fromJSON error; is.function
              branch can misclassify a plain function.
Also noted: no man/ pages, placeholder email, hand-written NAMESPACE vs roxygen.

--------------------------------------------------------------------------------
5. FIX ROUND #1 (user: "yes do this")
--------------------------------------------------------------------------------
Explanations given to user:
  - reactiveValues "additive vs exact": restore only wrote saved keys, never
    cleared stale keys present in receiving session -> result was the union.
  - is.function misclassification: a reactiveVal IS a function (read via rv()),
    so a literal function passed as data is treated as a reactiveVal and called.

Implemented:
  - Path traversal: .safe_filename() rejects separators, drive letters, ~, ..
  - unserialize gated behind allow_rds = FALSE (default), documented in a
    @section Security block.
  - max_size guard (default 50 MB) on capture and restore.
  - gzip compression (compress = TRUE) for file/rds payloads.
  - Exact reactiveValues restore: clear stale keys (set to NULL, since Shiny
    cannot truly delete a key) + new test.
  - Friendly fromJSON error wrapping.
  - Dropped stats:: (used names<-).
  - DESCRIPTION email -> gunjaca.iva@gmail.com.
  - roxygen-generated NAMESPACE + man/ pages.
  - Found subtle Rd bug: literal "%" in "~33%" is an Rd comment char that broke
    capture_state.Rd; replaced with "percent".
New tests: test-security.R, test-reactivevalues.R, test-edgecases.R.
Result: R CMD check Status: OK (no errors/warnings/notes); all tests green.

8. FIX ROUND #2 (user: "both" - fix decompression bomb + write README + thesis tex)
--------------------------------------------------------------------------------
Decompression bomb fix:
  - Rewrote .gzip_raw/.gunzip_raw to use gzfile connections (true gzip framing;
    memCompress(type="gzip") actually emits zlib, which gzfile passes through
    unchanged - that mismatch was caught during testing).
  - .gunzip_raw now streams in 1 MB chunks and aborts as soon as the running
    total exceeds max_size, so peak allocation is bounded.
  - Verified: a 508 KB payload claiming 500 MB aborts in ~18 ms without
    allocating the full output.
  - Added regression tests (bomb rejected; normal compressed payload still ok).
  - Made demo self-sourcing for source checkouts.
